# 📅 AURA Layer 초정밀 마이크로 실행 계획 (7/19 ~ 8/9)

## Phase 1: 기획 동결 및 데이터셋·API 스키마 설계 (7/19 ~ 7/23)

* **목표**: 시스템 아키텍처의 뼈대를 고정하고, 프런트엔드와 백엔드가 병렬 개발할 수 있도록 데이터 규격을 완벽히 정의.



### 7월 19일 ~ 20일: UI/UX 컴포넌트 Props 정의 및 피봇 확정

* **기획/PM**: 'AURA Layer' 최종 피봇 기획안 문서화 및 팀 내 싱크업.


* **프런트엔드**: 미리 정의된 **5대 정적 UI 블록**의 인터페이스(TypeScript Type) 설계.


```typescript
interface QuestionCardProps { id: string; question: string; options: ('YES' | 'NO' | 'DONT_KNOW')[]; }
interface ChecklistCardProps { items: { id: string; text: string; checked: boolean; }[]; }
interface LinkCardProps { title: string; url: string; institution: string; }
interface ActionCardProps { type: 'CALL' | 'SHARE'; payload: { phoneNumber?: string; script?: string; shareUrl?: string; }; }
interface AudioPlayerCardProps { textToSpeech: string; autoPlay: boolean; }

```


* **산출물**: Figma 와이어프레임(5개 컴포넌트 뷰), UI 블록 TypeScript 컴포넌트 명세서.

### 7월 21일 ~ 23일: 공공 데이터셋 수집 및 Gemini 구조화 프롬프트 엔지니어링

* **데이터/AI**: 공공누리(유형별 이용조건)를 준수하는 공공 안내문, 고지서, 행정 공문서 이미지 30종 확보 및 라벨링.


* **백엔드**: 정부24, 복지로 등 주요 10개 타깃 웹페이지의 핵심 이동 URL 및 필요 구비서류 모크(Mock) 데이터베이스 구축.


* **AI 엔지니어**: Gemini API가 임의의 텍스트를 뱉지 않고 정확히 `JSON` 규격(Structured Output)으로만 응답하도록 `System Prompt` 튜닝 및 `Response Schema` 강제 적용.


* **산출물**: AURA Public Task Dataset V1.0 (이미지 및 매핑 JSON), Gemini 시스템 프롬프트 확정본.

## Phase 2: 백엔드 엔진 구축 및 AI 컨텍스트 분석기 개발 (7/24 ~ 7/27)

* **목표**: 안내문 이미지를 입력받아 법적 안전성을 검증한 후, 정제된 과업 데이터 체인(Task Chain)을 생성하는 백엔드 API 완성.



### 7월 24일 ~ 25일: 멀티모달 입력 파이프라인 및 의도 분석기 구현

* **백엔드**: Fast API 기반 이미지 업로드 모듈 엔드포인트(`POST /api/v1/analyze`) 구현.
* **AI 엔지니어**: Gemini 멀티모달 API를 연동하여 이미지 내 행정 용어 추출 및 `user_goal`(이해, 신청, 납부 등) 분류 자동화.



### 7월 26일 ~ 27일: 공식 정보 매핑 및 책임 있는 AI 안전 레이어 구축

* **백엔드**: 분석된 안내문 키워드를 기반으로 모크 DB 내 정부24/복지로 공식 페이지 매핑 엔진 구현.


* **AI 엔지니어**: **안전성 검증 규칙(Safety Filter)** 레이어 추가.


* *규칙 1*: 결과 값 내 자격을 확정 짓는 단어(`"신청 대상자입니다"`)가 감지되면 문장을 자동 치환(`"신청 자격 가능성이 높으니 공식 확인이 필요합니다"`).


* *규칙 2*: 주민등록번호 등 민감한 개인정보 패턴(Regex) 감지 시 백엔드 단에서 즉시 마스킹 처리 및 파기.




* **산출물**: 통합 분석 API 서버, 데이터 개인정보 마스킹 모듈 코드.

### 7월 28일 ~ 29일: 5대 핵심 UI 블록 구현 및 웹 컴파일러 개발

* **프런트엔드**: TailWind CSS를 활용하여 고대비(High Contrast Mode), 거대 터치 영역, 아이콘-텍스트 병렬 표기를 준수하는 5대 컴포넌트 마크업 완성.


* **프런트엔드**: 백엔드 JSON 결과 배열의 `type` 필드(`"question"`, `"checklist"` 등)에 따라 적절한 컴포넌트를 순서대로 화면에 배치하는 동적 렌더링 엔진(UI Compiler) 구현.



### 7월 30일: 디바이스 네이티브 액션 및 Web Speech API 결합

* **프런트엔드**: `ActionCard` 연동을 위한 네이티브 기능 구현.
* `tel:02-XXX-XXXX` 프로토콜 기반 전화 걸기 레이어 구성(상단에 전화 시 말할 대사 스크립트 상시 렌더링).


* Web Share API를 활용한 모바일 기본 카카오톡/메시지 '가족 공유' 기능 구현.




* **프런트엔드**: Web Speech API(TTS) 엔진을 붙여 `AudioPlayerCard` 진입 시 문장 단위로 끊어 들을 수 있는 오디오 플레이어 연동.


* **산출물**: AURA Layer 웹 프런트엔드 MVP 빌드본, E2E 통합 테스트 통과 결과 보고서.

## 🚨 [위험 관리] 개발 중 돌발 상황 대처 매뉴얼 (Risk Mitigation)

1. **Gemini API의 일시적 레이턴시(속도 저하) 및 장애 발생 시**:
* *대응*: 데이터 수집 단계(Phase 1)에서 이미 정답 구조화 JSON이 매핑된 Mock API 모듈(`src/api/mock_v1.json`)을 내장해 둡니다. 대회 당일 라이브 시연 중 API 서버가 터지면 프런트엔드 스위치 변수를 즉시 `MOCK_MODE = true`로 전환하여 겉으로는 완벽히 작동하는 것처럼 방어합니다.


2. **공공 안내문 30종의 OCR 인식률 저하 발생 시**:
* *대응*: 글씨가 찌그러졌거나 오염되어 인식이 불가능한 안내문은 AI가 임의로 해석하지 않게 처리합니다. 프런트엔드에서 즉시 `QuestionCard`를 활용해 "글자가 흐려 식별이 어렵습니다. 이 문서가 '지방세 고지서'가 맞습니까?"와 같이 역으로 사용자 확인을 유도하는 인터랙션 흐름을 예외 코드로 구현해 실격 리스크를 차단합니다.
